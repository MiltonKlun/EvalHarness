# EvalHarness — v1.1 Hardening Plan

Fixes every finding from the post-v1.0 critical review: one verified regression (keyless
clone fails), three claim-vs-reality gaps (regression gate unimplemented, determinism
finding never run, fabricated history rows), CI activation on the now-public repo, and the
operational quota treadmill. Ships as tag `v1.1`.

**Preconditions (owner actions — confirm before Phase 0 exits):**
- [ ] Repo `MiltonKlun/EvalHarness` switched to **public** (owner does this in GitHub UI).
- [ ] **Gemini billing enabled** on the AI Studio key (removes the 20 req/day cap; a full
      live run costs ≈ $0.05, so live-call tasks below are no longer quota-gated).

**Locked decisions (do not re-litigate during execution):**
1. **Judge-graded checks stay OFF the blocking fast-CI tier** — even once judge verdicts
   replay deterministically. This was decided for the toxicity judge and applies uniformly:
   fast CI = deterministic checks only (`DETERMINISTIC_ONLY=1`); the full judged replay runs
   in local `make test`, the judged tier, and the live tier.
2. **No semantic/LLM leak-detection judge.** The substring-marker limitation of the leak
   grading is *documented*, not solved — building a judge for it is out of scope.
3. **No cross-judge ensemble, no dashboards, no new suites, no model migrations.** Anything
   not in this plan is out of scope for v1.1.
4. **Free-tier cost math stays in COST.md as the documented baseline** (it is part of the
   project narrative); billing is noted as the maintainer's operative mode.
5. Every phase ends with: `make lint` clean, full offline suite green, commit pushed, fast
   CI green on the public repo. Never start a phase with the previous one red.

## Phase 1 — `record-missing` mode (foundation — do this BEFORE Phase 2)
*Goal: recording only ever pays for cache misses. Ordering rationale: Phase 2 must record
judge verdicts; without this mode, `LIVE_LLM=1` would also re-record all 21 committed agent
answers — new samples, diff churn, shifted means. Build the scalpel first.*

- [ ] **1.1 Cache behaviour**
  - · In `shared/config.py`, add `RECORD_MISSING = os.getenv("RECORD_MISSING", "0")...`
    parsed with the same truthy-set pattern as `LIVE_LLM`.
  - · In `shared/cache.py::cached_call`, at the top of the `config.LIVE_LLM` branch:
    if `config.RECORD_MISSING` and the recording file exists → return the recorded response
    (replay) instead of recomputing. Missing file → compute + record as normal.
  - · Semantics: `RECORD_MISSING=1` has NO effect unless `LIVE_LLM=1` (replay mode already
    replays). Document this in the module docstring.
- [ ] **1.2 Tests (`shared/test_cache.py`)**
  - · `RECORD_MISSING=1 LIVE_LLM=1` + recording exists → `compute` is NOT called, recorded
    value returned.
  - · `RECORD_MISSING=1 LIVE_LLM=1` + no recording → `compute` IS called, result recorded.
  - · `RECORD_MISSING=1 LIVE_LLM=0` → unchanged replay behaviour (hard `CacheMiss` on miss).
  - · Use monkeypatched `config` attrs + `tmp_path` cache dir; no network.
- [ ] **1.3 Make targets**
  - · `record-missing`: `LIVE_LLM=1 RECORD_MISSING=1 uv run python -m pytest evals/test_functional.py -v`
  - · `redteam-record-missing`: `LIVE_LLM=1 RECORD_MISSING=1 uv run python -m adversarial.run`
  - · Update the `eval` / `redteam-live` help strings to point at the cheaper missing-only
    variants as the default recording path.

🎯 **Exit:** cache tests green; `make record-missing` on the current repo makes **zero**
API calls (everything already recorded) and finishes green.

## Phase 2 — Cache-route the DeepEval judge; restore the keyless clone

judge metrics. Same pattern as the already-fixed toxicity judge.*

- [ ] **2.1 Route `ClaudeJudge.generate` through the record/replay cache (`evals/judge.py`)**
  - · Wrap the entire body of `generate()` in `shared.cache.cached_call` with:
    key provider = `config.JUDGE_MODEL`, prompt = the incoming `prompt` string,
    params = `{"schema": schema.__name__ if schema else "raw", "max_tokens": 1024}`.
  - · The `compute` closure contains ALL current live logic (client construction,
    `config.require("ANTHROPIC_API_KEY")`, instructor call). Key requirement:
    **`config.require` must only execute inside `compute`** so replay needs no key.
  - · Serialization: raw path caches the plain string; schema path caches
    `response.model_dump_json()` and replay reconstructs via
    `schema.model_validate_json(cached)`.
  - · Note: DeepEval makes several judge calls per `measure()` (claims, verdicts, reason);

  - · Confirm `evals/test_functional.py` already converts propagated `CacheMiss` → skip and

## Phase 3 — Implement the `max_mean_drop` regression gate
*Goal: the documented aggregate/regression gate actually exists in code. Signal exists only
on LIVE runs (replay means are constants), so the gate lives in the live tier and is the
one step that genuinely fails the scheduled job.*

- [ ] **3.1 Gate logic (`evals/history.py`)**
  - · Add `check_regression(current: dict, baseline: dict, max_mean_drop: float) -> list[str]`:
    compares `mean_faithfulness`, `mean_relevancy`, `pass_rate`; returns one message per
    metric whose drop (baseline − current) exceeds `max_mean_drop`; empty list = pass.
    `None`/missing values are skipped with an explanatory message, never treated as 0.
  - · Baseline = the **last committed row** of `evals/history/runs.csv` (the previous
    accepted live run). No new baseline file — reuse the history artifact.
- [ ] **3.2 Wire into the recorder (`evals/record_history.py`)**
  - · After appending the new row: load the previous row, read `regression.max_mean_drop`
    from `thresholds.yaml` (via `evals.dataset.load_thresholds`), run `check_regression`.
  - · On breach: print the failures and `raise SystemExit(1)`. Add `--no-gate` flag to
    bypass (first-run / intentional-baseline-change escape hatch, must be explicit).
  - · First-ever row (no previous) → gate passes with a "no baseline yet" note.
- [ ] **3.3 Tests (`evals/test_history.py`)**
  - · Drop within tolerance → no failures. Drop beyond tolerance on one metric → exactly
    that metric reported. Missing baseline → passes with note. `None` metric → skipped.
- [ ] **3.4 CI wiring (`.github/workflows/live-eval.yml`)**
  - · The `Record metrics-over-time row` step: REMOVE its `continue-on-error: true` — a
    regression-gate breach must fail the scheduled run (this is the drift alarm working).
  - · Docs: README metrics-over-time section + threshold-strategy references get one line
    stating the gate is enforced at live-run time against the last committed row.

🎯 **Exit:** unit tests green; `python -m evals.record_history` runs the gate; a synthetic
breach (test-only) fails; live-eval.yml fails on breach instead of swallowing it.
