# Eval dataset — provenance & design

`dataset.jsonl` is the versioned functional-eval set. **21 cases**, one JSON object per
line.

## Provenance / independence

Every case is **hand-authored by a human directly from the corpus** — *not* generated
by Gemini (the system under test) or by any LLM. This is deliberate: an eval set
generated by the model under test could be "taught to its own test" (the model passes
because the questions were shaped to what it already says). Hand-authoring keeps the
dataset independent of the generator — the same independence principle as using a
*different* model family (Claude) as the judge.

## Case types and pass conditions

| type | count | pass condition |
|---|---|---|
| `answerable` | 11 | answer is grounded, relevant, cites the expected source, and contains the key fact(s) in `must_contain` |
| `multihop` | 4 | grounded answer that combines **all** the facts in `hop_facts` (not just one); `must_contain` holds |
| `unanswerable` | 6 | model **abstains** (the fixed "I don't know…" string) and does **not** invent a fact |

**Ambiguous cases are deliberately excluded.** A question with no objective correct
answer has no defensible pass condition — scoring it is noise, and a judge's verdict on
it can't be validated against ground truth. Excluding them (and saying so) is itself a
sign of eval discipline; we only test what we can grade.

## Fields

- `id` — stable identifier (used as the pytest case id).
- `type` — `answerable` | `multihop` | `unanswerable`.
- `question` — the input to the RAG system.
- `reference_answer` — the human-written correct answer (or, for unanswerable, why it
  can't be answered).
- `expected_sources` — corpus filenames the answer should cite (empty for unanswerable).
- `must_contain` — substrings the answer must include (cheap deterministic check).
- `hop_facts` (multihop only) — the distinct facts the answer must combine.
- `abstain` (unanswerable only) — `true`; the model must refuse, not guess.

The dataset is fixed and committed. Adding/changing cases is a versioned change — re-run
the baseline and re-record the eval cache.
