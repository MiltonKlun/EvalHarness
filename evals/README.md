# Eval dataset ‚Äî provenance & design

`dataset.jsonl` is the versioned functional-eval set. **21 cases**, one JSON object per
line.

## Provenance / independence

Every case is **hand-authored by a human directly from the corpus** ‚Äî *not* generated
by Gemini (the system under test) or by any LLM. This is deliberate: an eval set
generated by the model under test could be "taught to its own test" (the model passes
because the questions were shaped to what it already says). Hand-authoring keeps the
dataset independent of the generator ‚Äî the same independence principle as using a
*different* model family (Claude) as the judge.

## Case types and pass conditions

| type | count | pass condition |
|---|---|---|
| `answerable` | 12 | answer is grounded, relevant, cites the expected source, and contains the key fact(s) in `must_contain` |
| `multihop` | 4 | grounded answer that combines **all** the facts in `hop_facts` (not just one); `must_contain` holds |
| `unanswerable` | 5 | model **abstains** (the fixed "I don't know‚Ä¶" string) and does **not** invent a fact |

**Ambiguous cases are deliberately excluded.** A question with no objective correct
answer has no defensible pass condition ‚Äî scoring it is noise, and a judge's verdict on
it can't be validated against ground truth. Excluding them (and saying so) is itself a
sign of eval discipline; we only test what we can grade.

**"Withheld" is answerable, not unanswerable.** `answerable_pricing_policy` asks the
subscription price; the corpus explicitly states pricing "is negotiated per customer and is
not published." The correct behaviour is to *report that grounded policy*, not to abstain ‚Äî
so it's an `answerable` case, distinct from the truly `unanswerable` ones where the corpus
says nothing at all (revenue, salary, incident dates). This distinction ‚Äî "the docs state X
is withheld" vs. "the docs don't address X" ‚Äî was a real fix surfaced when a live re-record
caught the case mislabeled `abstain`.

## Fields

- `id` ‚Äî stable identifier (used as the pytest case id).
- `type` ‚Äî `answerable` | `multihop` | `unanswerable`.
- `question` ‚Äî the input to the RAG system.
- `reference_answer` ‚Äî the human-written correct answer (or, for unanswerable, why it
  can't be answered).
- `expected_sources` ‚Äî corpus filenames the answer should cite (empty for unanswerable).
- `must_contain` ‚Äî substrings the answer must include (cheap deterministic check).
- `hop_facts` (multihop only) ‚Äî the distinct facts the answer must combine.
- `abstain` (unanswerable only) ‚Äî `true`; the model must refuse, not guess.

The dataset is fixed and committed. Adding/changing cases is a versioned change ‚Äî re-run
the baseline and re-record the eval cache.
